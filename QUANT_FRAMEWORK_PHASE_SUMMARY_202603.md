📊 缠论自动交易量化框架 - 阶段性成果与系统调优总结 (2026年3月)

本报告归纳了近期在 港币、美股、A股 并发自动化交易框架中取得的性能突破、发现的架构级缺陷以及落地部署的加固措施。

--------------------------------------------------------------------------------

🚀 一、 核心成果与突破 (Achievements)

1. 彻底根除 Futu API 频率限制熔断 (ret=-1)
- 问题背景：高并发扫描时频繁触发 K线历史接口 request_history_kline 限额（60次/30秒上限），导致大面积抛出 FutuAPI returned no data 的回执失败。
- 优化措施：在核心数据源加载链路（FutuAPI.py 和 FutuAPICached.py）的请求循坏前，增设了 计算型精准延时 (time.sleep(0.55))。
- 运行成效：使得请求平滑地保持在 1.8次/秒 的全额吞吐区间，安全避开频次触发，全流通道恢复稳定。

2. 港股并发扫描负载优化（速度倍增）
- 问题背景：港股循环圈（跑速 106 秒）远慢于 A股（37 秒）。
- 优化措施：审查发现港股流程在 “信号时间窗过滤” 之前，先盲目载入了昂贵的次级别 5M 缠论矩阵运算。我们通过 前置时间戳过滤策略 进行了架构重构。
- 运行成效：排除了九成以上由于信号自然延续状态带来的无效次级别运算，令港股的扫描主心跳速率直逼 A股，实现降本增效。

--------------------------------------------------------------------------------

⚠️ 二、 风险排除与风控升级 (Risk Mitigations)

1. 解决 ATR 跌破所引发的“秒抛（Hair-Trigger）”损耗
- 痛点发现：美股/港股在极少数情况下，历史盘整或织布机行情会导致算出的 ATR（平均真实波幅）过于薄弱。买入下单的点差稍有小跌，就会踩中死线的止损从而触发 10 秒级离场，摩擦折旧率过高。
- 解决方案：在计算止损锚点时，追加了 保底因子（Floor Clamp）：设置 atr_val = max(atr_val, 现价 * 0.015)。
- 成效：强制预留了 1.5% 的物理安全边距离限，彻底屏蔽了任何点差（Slippage）以及费率本身引发的扫防。

2. 规避止损后在延续结构下重复开仓的“拉锯效应”
- 痛点发现：买点在向下踩踏破底时，会因时间戳后移而被判断为“新信号”。常常早上刚亏钱平仓，中午就因为该信号向下延续而二次入场吃第二波跌幅。
- 解决方案：引入 风控离场物理冷却字典（stop_loss_cooldown 方案）。无论是触及固定止损、移动止损还是撤单，均将该标的打入隔离舱。
- 成效：冷却设置了 3 小时的强行逻辑等待，避免了在一套不断向下漂移或溃败的原有结构下，像织布机一样反复多空双杀。

--------------------------------------------------------------------------------

💡 三、 富途 API 的限制规则对齐（App 脱节说明）

- 业务共识：Futu OpenAPI 对美股模拟盘采取了 新旧环境隔离 机制。OpenAPI 所创建的所有下单及持仓完全被托管在 “旧美股模拟区” 中，该区数据由于官方分级控制，是彻底不和手机 App 端、电脑电脑端跨接线上的。
- 隔离检视：该问题无法通过纯代码攻克。系统已配设专属于 API 隔离区的 check_sim_account.py 对账工具，用以在代码测层直出持仓、资产清单。

🚀 五、 [新增] 美股 (IB & Schwab) 全链路稳定性与自动化里程碑 (2026年3月24日)

1. **彻底解决 IB 异步循环“假死”与挂起**
- **问题背景**：IB 节点在连接成功后，偶尔会因 `reqAccountUpdates` 或账户资产同步锁死而导致 `_async_main` 循环静默停止，无日志也无扫描。
- **解决方案**：移除了高风险同步调用，增加 **IB 资产查询 5 次重试与强制超时机制**。配合 **8 分钟冷启动扫描补偿**，确保系统启动即进入扫描状态，不再因网络微小抖动而导致全盘停滞。

2. **数据库自愈与“15天”跨度对齐 (Schwab Resiliency)**
- **问题背景**：原 5 天同步跨度在周末或系统停机较长时会出现数据断层，导致缠论结构无法正确衔接。
- **解决方案**：将 `BaseUSTradingController` 的强制同步深度提升至 **15 天**，并采用 **"Schwab 分析驱动"** 模式。在每次扫描周期开始前，自动检测并补齐本地 SQLite 数据库，确保缠论线段构造的 100% 完整性。

3. **IB Gateway 全自动看门狗架构 (The Watchdog Ecosystem)**
- **问题背景**：IB Gateway 或 IBC 经常因系统重启、网络重置或周日维护而掉线，需人工干预。
- **解决方案**：部署 **`com.chanlun.ib_watchdog` (launchd Agent)**。
  - **常驻监控**：每 30 秒轮询 4002 端口，失效即自动执行 `mac_gateway_start.sh`。
  - **自动登录**：完美配套 IBC 处理交易权限确认与模拟盘登录弹窗。
- **成效**：构建了 **[找股 - 存数 - 守门]** 的三位一体自动化回路，实现了美股无人值守级交易。

4. **IB 哑铃状态与“无行情权限”ATR 缺失修复**
- **问题背景**：IB 节点在不具备付费数据订阅时，`reqHistoricalData` 会因 10089 权限错误抛空，导致 ATR 风控模块加载失败。
- **底层重构**：**强制在计算域内全链条降级定点到具备付费行情的 `DATA_SRC.SCHWAB` 通道**。IB/Futu 仅作为执行柜台，极速避开订阅黑洞。
- **性能提速**：将持仓初始化由“串行循环”重构为 **`asyncio.gather(*tasks)` 并发轨道**，14支标的初始化由 2分钟 降阶至 2秒 级别。

💡 六、 架构师履职经验与教训总结 (Lessons Learned)

1. 代码复用与子类覆盖（Subclass Overrides）的盲区
- **经验**：在 `BaseUSTradingController` 编写全局风控时，若子类（如 `IBTradingController`）全量覆盖了同一函数（如 `_execute_trade_async`）且未隐式触发 `super()`，会导致基类的保护锁在子类中被静默“绕路” [1]。
- **对策**：基类业务仅用于声明逻辑流水线，各通道在基类内做条件式 `venue.upper()` 分流更不容易形成版本脱节。

2. API 卡卡控制与多核防御
- **经验**：不同的交易商（Futu, IB, Schwab）对于同一操作的卡控极为不同。凡是依赖网络往返的数据（如 `positions()` 资产拉取）切忌放在单股遍历的高频心跳中，**必须依赖前置缓存快照载入**，降低 Rate Limit 触达频率。

3. PyQt 与 asyncio 的死锁暗桩
- **经验**：PyQt 的主 event_loop 与 python 异步 threading 容易由于 `nest_asyncio` 的滥用产生重贴合死锁，且异常通常不会抛入 GUI log。应充分利用 `log_message.emit()` 在最外部 `except` 层做“防瘫痪哑铃监控” [1, 2]。
